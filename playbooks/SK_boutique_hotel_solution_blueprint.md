# SK boutique hotel – Solution Blueprint (Markdown Version)

## 1. Product Definition

This solution is designed as a future-ready hotel website and operations platform for **SK boutique hotel**, combining:

- a **public-facing website** with premium branding, room showcase, branch information, offers, contact, and membership capture
- an **admin/staff portal** for content, room inventory visibility, lead handling, manual booking, deposit confirmation, and analytics
- a **future-ready architecture** prepared for OTA integration and automated online payment in later phases

Phase 1 is intentionally built as a **manual booking operations model**, not a fully automated online booking engine.

## 2. Phase 1 Business Goal

In phase 1, visitors can:

- browse rooms and hotel content
- check room availability
- submit hold-room requests
- sign up as members
- receive offer/marketing emails if consent is given
- view their request and booking history after login

Internal users can:

- manage branches, floors, physical rooms, and room types
- manage multilingual website content (VI/EN)
- receive availability requests in real time and by email
- create and manage manual bookings
- issue deposit requests through dynamic VietQR
- verify payment proof manually
- confirm bookings and issue confirmation email + PDF

## 4. Key Business Principles

1. **Public display is based on room type**
2. **Operations are managed by physical room**
3. **Manual workflow in phase 1, scalable architecture underneath**
4. **Multilingual content in VI/EN**
5. **Premium modern UI with a restrained boutique hospitality feel**
6. **Audit logs and permission control are required**
7. **Booking and payment automation are future modules, but schema must be prepared now**

## 5. Agreed Recommendations

The following recommendations were accepted and should be treated as implementation principles:

1. Public-facing UI should feel refined and premium, while backend logic must behave like a real hotel operations system.
2. Although phase 1 limits one booking to one room in the UI, the core schema should remain extensible for future multi-room booking.
3. Payment proof should support both a secure email link and member portal upload.
4. Booking confirmation should support **email + PDF** in phase 1.
5. Floor view in phase 1 should use a **list/grid by floor**, without requiring a complex graphical floor map.

## 6. Functional Highlights

### Content & Branding
- hero carousel / sliders
- modular landing page sections
- room gallery and detailed room content
- SEO title/meta management
- blog/news with approval workflow
- seasonal/campaign banners in simplified mode

### Hotel Structure
- 2 branches initially
- room types prepared even if client data is not fully available at project start
- physical room management by branch and floor
- maintenance and blocked room states supported in phase 1

### Availability & Booking
- visitor can request room check or room hold
- SLA for first response: **30 minutes default, editable**
- hold expiry before deposit: **30 minutes default, editable**
- booking expires automatically if deposit is not confirmed in time
- system suggests suitable physical rooms, staff may override if authorized

### Pricing
- `base_price`
- `weekend_surcharge`
- `manual_override_price`
- simple promotion support in phase 1
- public price can be toggled:
  - show “From xxx / night”
  - or hide price and show CTA only

### Membership & CRM
- account creation can begin from availability request submission
- separate marketing consent checkbox, default checked
- email verification is not forced for availability requests
- first actual login requires password setup
- customer can see request/booking history after login
- notes per lead/member supported at a basic level

### Payment / Deposit
- each branch has its own bank account
- dynamic VietQR generated by amount + booking code
- customer uploads payment proof via secure link or portal
- staff manually verifies incoming transfer
- confirmation email + PDF are issued after payment verification

### Analytics
- visit and click tracking
- room interest and branch interest tracking
- traffic source tracking
- dashboard with filtering by date / branch / room type
- ability to identify customers who repeatedly viewed rooms after they become known users

## 7. User Roles

### Guest / Member
- browse content
- submit room check or hold request
- manage profile and history after login
- upload payment proof
- receive booking and marketing emails

### Staff
- receive and process requests
- add notes and communicate with guests
- create bookings if permitted
- confirm payment if permitted
- edit content only if permitted

### Manager
- manage assigned branches
- review bookings, requests, and analytics within scope
- supervise staff within assigned branches

### Admin
- manage overall business data, content, approvals, users, and operational settings
- approve blog/news workflow

### System Admin
- manage technical settings, monitoring, logs, permissions, and full system supervision

## 8. Target Tech Stack

- **Frontend:** Next.js App Router + Tailwind CSS
- **Backend/Data:** Supabase PostgreSQL + Auth + Storage + Realtime
- **Email:** Postmark or Resend
- **Files/PDF:** Storage + server-side PDF generation
- **Deployment:** Vercel + Supabase
- **Background jobs:** hold expiry, booking expiry, SLA reminders, scheduled notifications
